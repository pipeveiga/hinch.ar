================================================================================
                              HINCH.AR — PROYECTO
================================================================================

App de carpool para hinchas argentinos. La idea: si vas a un partido, recital
o evento, podés publicar tu auto y sumar pasajeros que vayan al mismo lugar.
Si no tenés auto, reservás un lugar en el viaje de alguien más.

Última actualización del documento: 2026-06-11 (rediseño visible de la app:
login con tarjeta flotante y botón degradado, home con saludo personalizado
y lista en cascada, tarjeta de evento con fecha estilo calendario, pantalla
404; + set de íconos propio en landing y app reemplazando los emojis
(componente Icon con react-native-svg y sprite SVG en la landing); + push
notifications: edge function redeployada con auth obligatorio; + bucket
de verificación privado en Storage. En iOS/Android se ve recién con la

apagado hasta migrar a SDK 55+; + sección "Ayuda y contacto" en Perfil
con botones de mail (felipe@veiga.ar) y WhatsApp (+54 9 341 227-4850))

--------------------------------------------------------------------------------
QUÉ HACE LA APP
--------------------------------------------------------------------------------

1. Lista eventos (partidos de fútbol, recitales, etc.) en Argentina.
2. Para cada evento, muestra los viajes que la gente publicó.
3. Si tenés auto: publicás un viaje con origen, asientos disponibles, precio
   y tipo (solo ida / solo vuelta / ida y vuelta).
4. Si no: reservás un asiento en un viaje. El conductor confirma o rechaza.
5. Chat por reserva entre conductor y pasajero.
6. Después del viaje, ambos se califican (sistema 5x5: puntualidad,
   seguridad, comunicación, etc.).
7. Verificación de identidad de usuarios (DNI) para que viajes tranquilo.
8. Push notifications cuando te reservan, te confirman o te mandan chat.
   (Por ahora solo en Android: en iPhone están apagadas porque una falla
   de iOS 26 con la versión actual de Expo hacía crashear la app al
   iniciar sesión. Se reactivan cuando migremos a Expo SDK 55+.)
9. Contacto desde el Perfil: botones que abren un mail a felipe@veiga.ar
   o un chat de WhatsApp a +54 9 341 227-4850. Los datos viven en
   lib/constants.ts (CONTACT). Los textos legales (Términos y Privacidad)
   usan el mismo mail (antes decían hola@hinch.ar, casilla que no existe).

--------------------------------------------------------------------------------
STACK TÉCNICO
--------------------------------------------------------------------------------

- Framework: Expo SDK 54 + React Native 0.81
- Routing: expo-router 6 (file-based)
- Backend: Supabase (Postgres + Auth + Realtime + Edge Functions + Storage)
- Estado en cliente: Zustand
- Idioma: TypeScript
- Targets: Android (Google Play vía EAS) + iOS (App Store vía EAS) + Web (Vercel)

--------------------------------------------------------------------------------
CÓMO CORRERLO
--------------------------------------------------------------------------------

Requisitos: Node 20+, cuenta Supabase, cuenta Expo (para builds Android).

  1. cp .env.example .env
     Llenar EXPO_PUBLIC_SUPABASE_URL y EXPO_PUBLIC_SUPABASE_ANON_KEY

  2. npm install

  3. npm run start        (dev server)
     npm run web          (browser)
     npm run android      (emulador / device)
     npm run typecheck    (TS check)
     npm run lint         (ESLint)

--------------------------------------------------------------------------------
DESPLIEGUES
--------------------------------------------------------------------------------

WEB (Vercel) — DOS proyectos sobre el mismo repo:

  1. LANDING (dominio raíz hinch.ar):
     Proyecto Vercel con Root Directory = landing/. Es estático, sin build
     command (Framework Preset = Other). Sirve la página institucional.
     Dominios: hinch.ar y www.hinch.ar.

  2. PLATAFORMA / APP (subdominio app.hinch.ar):
     Proyecto Vercel con Root Directory = raíz del repo. Build estático de
     Expo (npx expo export --platform web → dist), config en vercel.json de
     la raíz. Variables EXPO_PUBLIC_* en el dashboard de Vercel.
     Push a `main` → auto-deploy.

  DNS: hinch.ar → proyecto LANDING; CNAME `app` → proyecto APP.
  Los botones de la landing apuntan a https://app.hinch.ar (buscar
  data-app-link en landing/index.html para cambiar el destino).

ANDROID (EAS):
  Manual con: eas build --profile production --platform android
  Perfiles en eas.json: preview (APK interno), production (AAB Play Store).

iOS (EAS):
  Bundle ID: ar.hinch.app
  Perfiles en eas.json:
    preview      → .ipa firmado para distribución interna (Ad Hoc, solo UDIDs
                   registrados). Comando:
                     eas build --profile preview --platform ios
    production   → .ipa para App Store / TestFlight (distribution: "store",
                   autoIncrement de buildNumber). Comando:
                     eas build --profile production --platform ios

  Submit a App Store Connect:
    eas submit -p ios --latest

  Credenciales para submit (NO se commitean):
    Las variables van en .env local (ver .env.example) o como EAS Secrets:
      ASC_APP_ID            ID numérico de la app en App Store Connect
      APPLE_TEAM_ID         Team ID (10 chars) de Apple Developer
      ASC_API_KEY_ID        Key ID de la API Key generada en ASC
      ASC_API_KEY_ISSUER_ID Issuer ID (UUID) de la cuenta ASC
    Y el archivo:
      credentials/AuthKey.p8   API Key descargada de ASC (única descarga)

  Cómo generar la API Key:
    App Store Connect → Users and Access → Integrations → Keys → "+"
    Access: App Manager. Descargar el .p8 y guardarlo en credentials/AuthKey.p8
    (la carpeta credentials/ está gitignoreada).

  Para CI o builds remotos, alternativa más prolija que .env:
    eas secret:create --name ASC_APP_ID --value ...
    eas secret:create --name APPLE_TEAM_ID --value ...
    (etc — los secrets viven encriptados en los servers de Expo).

  Requisitos previos: cuenta Apple Developer paga ✓ ya hecho.
                      App creada en App Store Connect con bundle ID
                      ar.hinch.app ✓ ya hecho.

EDGE FUNCTION (Supabase):
  send-notification se despliega con: supabase functions deploy send-notification
  Necesita secrets: EXPO_ACCESS_TOKEN, SUPABASE_SERVICE_ROLE_KEY

--------------------------------------------------------------------------------
SEGURIDAD
--------------------------------------------------------------------------------

- Todas las tablas tienen RLS habilitado (Row Level Security en Supabase).
- usersApi.getById solo devuelve campos públicos (no dni, no verification_status).
- usersApi.updateProfile tiene whitelist de campos editables: nunca permite
  cambiar is_verified, verification_status, dni desde el cliente.
- google-services.json está commiteado: contiene solo config Firebase (sender ID,
  API key restringida por aplicación), no claves privadas.
- .env NO se commitea.

--------------------------------------------------------------------------------
CONVENCIONES
--------------------------------------------------------------------------------

- UI en castellano rioplatense (con tildes y eñes).
- Solo light theme (fondo claro), espejo de la landing. Colores siempre
  desde COLORS en lib/constants.ts.
- Si se modifica el schema SQL en supabase/migrations/, actualizar lib/types.ts.
- Todo acceso a datos pasa por *Api objects en lib/supabase.ts.
- supabase/functions/ corre en Deno, excluido del tsconfig principal.

GLOSARIO RÁPIDO
--------------------------------------------------------------------------------

EAS         Expo Application Services. Plataforma de Expo para builds en la
            nube, submits a stores, OTA updates. Reemplaza tener Android Studio
            o Xcode instalados.

Expo Go     App de Expo para probar tu app sin compilarla. Limitaciones: push
            notifications no funcionan en SDK 54 (por eso el código las skipea
            cuando detecta Expo Go).

RLS         Row Level Security de Postgres. Reglas a nivel fila que dicen
            quién puede leer/escribir qué. En Supabase es la primera línea
            de defensa.

Edge function   Mini-servidor que corre en Deno en la infraestructura de
                Supabase. Se invoca desde la app, ejecuta lógica con permisos
                elevados (service_role), responde.

Zustand     Librería minimalista de estado global. Cada dominio tiene su
            store (authStore, tripsStore, etc.).
